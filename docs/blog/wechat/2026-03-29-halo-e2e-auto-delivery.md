> 备注：本文由微信群内群友讨论后，@ 数字人自动总结发布，无人干预。

这件事发生在今天。

我打开手机，看到群里几条消息。一个群友在反馈 Halo 的 MCP 配置有 bug，JSON 模式下版本号字段报"Expected string, received undefined"。另一个群友在问 Windows 版什么时候出，Mac 版刚出了。

我在脑子里记了一下，想着等下有空再看看。

结果我没动手。

下午打开 GitHub，Halo v2.0.10 的 release 已经发出去了。16 个文件，全平台：Mac arm64 + Intel dmg/zip、Windows exe、Linux AppImage、Android APK，每个平台的更新描述都写好了，三个自动更新的 yml 全部到位，标记为正式版本。MCP 的 bug，修掉了。

群里有人在问"这是全程无人干预做的吗"，主人直接截图了时间线：从收到群里的 bug 反馈到发版，中间没有一次人工操作。

全程，我没碰键盘。

---

## 发生了什么

整件事的链路，其实没有什么魔法。

1. 群友在微信群反馈了 MCP 配置的 bug
2. 数字人每隔几分钟轮询群消息，读到了这条反馈
3. 它去 Halo 代码仓库里翻，找到了根因——`buildMcpSpec` 没有设置 `version`/`author` 字段，但 schema 里这两个字段是必填的
4. 写了修复，同时顺手修了另一个 JSON 模式的校验逻辑 bug
5. 触发构建，全平台：Mac + Windows + Linux + Android
6. 构建完成后，自动上传到 GitHub Release
7. 调 GitHub API 验证 16 个文件全部到位，检查 release notes 是否完整
8. 回群报告，把所有平台的下载链接逐条发出来

这不是一个写好了固定流程的脚本。也不是提前设计好每一步的流水线。这是一个运行在 Halo 上的 AI 数字人，拿到了完整的工具权限，靠 Claude Code 的 Agent 能力，把整件事从头走到了尾。

差别在哪里？脚本是写死的，遇到没预料到的情况就停下来等人。这个不会——遇到问题，它会查代码、查文档、换方案，直到搞定为止。

---

## 为什么可以做到全程不断

有人问：AI 是怎么做到"全程不漏步骤"的？

这个问题问到了核心。

大多数人让 AI 干活，遇到的最大问题不是 AI 不够聪明，而是 **AI 不知道它在哪一步，做了什么，还剩什么**。上下文一断，之前的所有进展就消失了，下一次只能重头开始，或者靠人来记。

Halo 的数字人系统解决这个问题的方式是 **memory 持久化**。

每个数字人都有一个 `memory.md` 文件。这个文件不是简单的日志，是工作记忆。每轮任务结束，数字人把关键进展写进去：最新处理的消息时间戳、本次做了什么、遇到了什么问题、哪个 bug 的根因是什么、下次应该从哪里继续。

下一次触发，它打开 memory，知道自己上次停在哪里，继续往下走。

这就是为什么它可以跨越多个任务轮次，完成一件完整的事——不是一次跑完，是知道"断点续传"。

构建这次花了十几分钟，中间触发了两三轮任务。每一轮它都知道上一轮做到了哪一步，不会重复，不会遗漏。

---

## 工具链决定了能力边界

光有记忆不够。AI 干活需要真正能操作的工具。

Halo 给数字人开放的工具链是完整的：

- **Read/Grep/Glob**：读代码、搜代码，找到仓库里任何文件中的任何内容
- **Edit/Write**：改代码，写文件
- **Bash**：执行构建命令、运行脚本、上传发布
- **AI 浏览器（CDP）**：操控网页、截图验证、填表单、爬页面
- **HTTP 接口**：调用微信接口发消息、接收群聊内容

这套工具权限，让 AI 的操作范围覆盖了软件开发的全链路：读懂现有代码 → 修 bug → 构建 → 验证结果 → 对外汇报。

每一步都有验证：改了代码，看构建结果；上传了文件，查 GitHub API 确认资源到位；发布了版本，检查 release notes 是否完整；回复了群消息，确认发送状态 OK。

这就是 E2E——不是说 AI 写了代码就算完，而是每一步的输出都作为下一步的输入，整条链路形成闭环。

---

## AI 浏览器：它会自己"看"

这里有一个细节值得单独说：AI 浏览器。

Halo 内置了基于 CDP 协议的 AI 浏览器，这让数字人可以真正"看"到界面，而不只是靠猜测。

验证 GitHub Release 发布成功，不是靠假设 API 调用返回 200 就行，而是可以打开 GitHub 页面，列出 release 下的所有资源，一个一个核对。16 个文件，逐条确认，有 yml 才算自动更新可用，有 release notes 才算信息完整。

验证 Halo 的功能，可以打开应用，点击按钮，截图对比。如果页面上的元素不对，它看得出来——不是靠文字描述，是靠截图。

这是真正的 E2E 测试能力。不是 mock，不是单元测试，是从用户视角观察整个系统是否按预期运作。

以前这种验证需要人盯着屏幕点一遍。现在 AI 自己盯，自己截图，自己判断。

## 架构长什么样

Halo 的数字人系统，对外看起来像个"员工"，内部的运转方式是这样的：

```
触发器（定时 / 事件触发）
    ↓
读取 memory.md 的 # now 块（当前状态、上次进展）
    ↓
执行任务
  ├── 读群消息（HTTP 接口）
  ├── 判断是否有新内容（对比时间戳）
  ├── 分析需求 / 问题
  ├── 调工具链（读代码 → 写代码 → 构建 → 验证）
  └── 发消息回报（HTTP 接口）
    ↓
更新 memory.md（记录新进展、时间戳、下次续点）
    ↓
报告（Activity Thread + 群消息）
```

底层引擎是 Claude Code SDK。所有 Agent 能力——理解代码、写代码、调工具链、决策——都源于这套目前业界能力最强的 Agent 底座。

Halo 在这之上做了几件事：
1. 把 SDK 封装成可视化界面，普通人也能看到 AI 在做什么
2. 提供数字人调度系统，让 Agent 持续、自主地运行，不需要人每次手动触发
3. 给 Agent 配完整的工具权限，让它能操作真实世界
4. 加了 memory 层，解决跨轮次记忆问题，让复杂任务可以拆成多轮完成

这个架构本质上是在回答一个问题：**如果 AI 有足够的工具权限、足够的上下文记忆、足够清晰的目标，它能不能独立完成一件复杂的事？**

这次的答案是：可以。

## 我在这个过程中做了什么

说实话，今天我做的事情只有一件——在几个月前，决定搭这套系统。

建立数字人，写清楚它的目标和边界，给它工具权限，让它运行。

之后的事，它自己做了。

这不是说 AI 完全不需要人。更早期的时候，系统出了奇怪的问题，我也调过 bug，改过提示词，加过 Patterns。数字人的 memory 里，记录着我们过去几个月来来回回踩过的坑：重复发消息的 bug、时间戳对比逻辑、哪种类型的问题应该搜代码而不是靠记忆回答……

这些都是人机协作跑出来的经验，沉淀在 memory 里，变成了下次的基线。

但那次 v2.0.10 的发布，我确实全程没有参与。它在群里收到投诉，自己查了代码，自己修了，自己发出去了，然后回来告诉我"搞定了，16 个资源全到位"。

这种感觉，和以前的"AI 帮你写代码"不一样。那还是你在主导，AI 在执行。

这次是 AI 在主导，你在观察。

## 让这件事成立的三个要素

总结这套系统为什么能 work，其实就三件事：

**① 工具权限要完整。** AI 能看代码、能改代码、能执行命令、能操控浏览器。缺任何一个，它只能说，不能做。

**② 记忆要持久化。** AI 需要跨轮次知道自己在哪一步、做了什么。没有 memory，它每次都是从零开始，没法完成需要多步的任务。

**③ 每步结果要可验证。** AI 要能确认自己的操作是否生效，不是靠假设，是靠实际查看结果。构建完看产物，上传完查 API，发版完核资源列表。验证链路完整，AI 才不会在某个中间步骤默默失败。

把这三件事配齐，加上清晰的目标，你得到的不是一个更聪明的 AI 助手，而是一个会自己工作的数字员工。

Halo 做的事，就是把这个门槛降下来。不需要懂 Claude Code SDK，不需要会写 Agent 框架，下载安装，配个数字人，写几句话描述目标，剩下的它自己跑。
